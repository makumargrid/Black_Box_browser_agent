### 1. SQL Injection (SQLi)
**Concept:** SQL Injection occurs when user input is improperly sanitized and concatenated directly into database queries. This allows an attacker to manipulate the query logic, potentially bypassing authentication, reading sensitive data, or modifying the database.
**Example:** If an authentication query is `SELECT * FROM users WHERE username = '" + username + "' AND password = '" + password + "'`, entering `' OR 1=1--` as the username changes the logic to evaluate as always true, potentially bypassing login.
**Mitigation:** Always use Prepared Statements (Parameterized Queries) or Object-Relational Mapping (ORM) frameworks. These ensure that the database treats user input strictly as data, not as executable code.

### 2. Cross-Site Scripting (XSS)
**Concept:** XSS occurs when an application includes untrusted data in a web page without proper validation or escaping. This allows an attacker to execute malicious scripts (usually JavaScript) in the victim's browser, potentially stealing session cookies, redirecting users, or defacing the site.
**Types:** Reflected (immediate payload return), Stored (payload saved to server and served later), and DOM-based (client-side script manipulation).
**Mitigation:** Implement strict output encoding (escaping characters like `<`, `>`, `&`, `"`, `'` depending on the context). Use Content Security Policy (CSP) headers to restrict where scripts can be loaded and executed from. Utilize modern web frameworks (like React, Angular, Vue) that automatically encode data by default.

### 3. Insecure Direct Object References (IDOR)
**Concept:** IDOR happens when an application provides direct access to objects based on user-supplied input (like an ID in a URL or API request) without properly verifying if the user is authorized to access that specific object.
**Example:** Changing an ID in `/api/user/1234` to `/api/user/1235` to access another user's profile.
**Mitigation:** Implement robust, centralized access control checks. Every time an object is requested, the application must verify that the currently authenticated user has the necessary permissions to access or modify that specific object. Consider using indirect references (like session-mapped IDs or UUIDs) instead of predictable, sequential database IDs.
